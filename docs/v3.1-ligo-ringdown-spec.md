# v3.1 deferred — Real LIGO ringdown fitter integration (spec)

**Status:** deferred from v3.1.  Multi-week effort; out of scope for the current sprint.

This document is the spec for when we eventually implement it, so
nobody has to re-derive the design.

## Goal

Take a real LIGO/Virgo binary-black-hole merger event from the
public GWOSC catalog (e.g., GW150914), fit its ringdown phase to a
linear superposition of Kerr quasinormal modes, and return the
posterior on `(M_final, a_final/M)` plus per-mode amplitude/phase
estimates.  Display a corner plot in the existing
`spacetime_lab.waves` page.

## Why it's a multi-week project (honest scope)

1. **Bilby dependency**: ~50 MB install, pulls in `dynesty`, `pymultinest`,
   `lalsuite`, `pyfftw`.  Backend Render free-tier won't fit; we'd need
   a separate worker container.
2. **Data**: GWOSC strain files are HDF5, ~MB each per event per detector.
   Caching strategy needed (one-time download → S3/R2 bucket → backend pulls
   on demand).
3. **Compute**: One MCMC posterior takes 30-120 min on 4 CPU cores per event.
   We can't run it on every page load.  Architecture: precompute posteriors
   for the catalogued events (offline job), serve cached results.
4. **Detector-frame vs source-frame**: redshift correction needs cosmology
   choice; spec explicitly which (Planck18 default) to keep results
   reproducible.
5. **Frontend**: corner plots need Plotly or `vega-lite`; we currently
   ship neither (deliberately — see ROADMAP-v3 stack notes).
6. **Validation**: results must agree with Berti-Cardoso-Starinets 2009
   table and with Carullo et al. 2019 GW150914 ringdown analysis to
   within published error bars.  Non-trivial regression burden.

## API design (when implemented)

```
GET /api/waves/ringdown/events
  → { events: [
        {id: "GW150914", date: "2015-09-14", M_final_median: 62.0, ...},
        {id: "GW170817", ...},
      ] }

GET /api/waves/ringdown/{event_id}/posterior
  → {
      event: "GW150914",
      sampler: "dynesty",
      n_samples: 4000,
      params: {
        M_final:   [...],   // mass posterior chain
        a_over_M:  [...],   // spin posterior chain
        amp_220:   [...],   // dominant mode amplitude
        phi_220:   [...],   // dominant mode phase
      },
      summary: {
        M_final_median: 62.0,
        M_final_low:   58.5,
        M_final_high:  65.7,
        ...
      },
      detector: "H1L1",
      ringdown_start_ms_after_merger: 3.0,
    }
```

## Phased implementation plan (when we pick this up)

### Phase A — offline pipeline (1 week)
- [ ] Bilby install in a dedicated `worker/` container with its own
      requirements file (don't pollute backend reqs)
- [ ] `worker/scripts/precompute_ringdown.py` — accepts event_id,
      pulls strain from GWOSC, runs Bilby ringdown likelihood, dumps
      posterior chain as Parquet
- [ ] R2 bucket layout: `s3://spacetime-lab-ringdown/{event}/posterior.parquet`
- [ ] Run script for GW150914, GW170817, GW190521 as initial set
- [ ] Cross-check vs Carullo et al. 2019 within 1σ

### Phase B — backend exposure (1 week)
- [ ] `backend/app/routes/waves.py` — new module for the waves
      subpackage REST surface (currently no waves routes exist)
- [ ] `/api/waves/ringdown/events` lists what's cached
- [ ] `/api/waves/ringdown/{event}/posterior` reads Parquet from R2,
      returns JSON-serialised chain (limit 4k samples for payload)
- [ ] Smoke tests in `tests/test_api_waves_ringdown.py`

### Phase C — frontend visualization (1 week)
- [ ] Add Plotly to package.json (or `chart.js` if Plotly bundle too big)
- [ ] `pages/Waves.jsx` — new route currently absent (waves doesn't have
      a UI page yet; this would be the first)
- [ ] `components/CornerPlot.jsx` — 4×4 corner plot of {M, a/M, amp, phi}
- [ ] `components/PosteriorComparison.jsx` — overlay measured vs predicted
      Kerr QNM (from existing Phase 5 / v1.2 wrapper)

### Phase D — validation + docs (3 days)
- [ ] Notebook `notebooks/16_ringdown_inference.ipynb` reproducing the
      headline plot
- [ ] CHANGELOG entry, ROADMAP-v3 sync
- [ ] Cite Carullo, Del Pozzo, Veitch 2019 + LIGO Open Science Center

## Risk register

- **R1** Bilby version drift breaks reproducibility → pin to a specific
  Bilby tag in `worker/requirements.txt`
- **R2** GWOSC API rate limits → mirror necessary strain files in R2
- **R3** R2 bandwidth cost on heavy traffic → cap response payload
  to first 4k posterior samples; offer Parquet download for full chain
- **R4** Posterior samples can be ~10 MB each → check Render free
  tier egress limits before promising production deploy
